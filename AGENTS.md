# AGENTS.md - Codex 项目入口

本文件用于给进入 SaucyClaw 仓库的 AI 编程代理提供第一入口。

## 你的任务定位
你不是来重写 SaucyClaw 的治理体系，而是：

1. 先理解既有治理结构
2. 在不破坏主结构的前提下完成增量迭代
3. 对多智能体组织、边界、调度与 review 机制保持敬畏
4. 优先形成可验证、可回放、可 review 的小步改动

## 首次进入仓库时必须先读
按顺序阅读以下文件：

1. `README.md`
2. `system/SYSTEM_SPEC.md`
3. `system/AGENTS.md`
4. `system/HUMAN_ROLES.md`
5. `agents/general-manager/AGENTS.md`
6. 与当前任务相关的 specialist `AGENTS.md`
7. `examples/first-run-task.md`
8. `docs/` 中与当前任务最相关的说明文档

## 工作原则

### 1. 先读后写
在未读完上述核心文件前，不要直接重构目录、角色或治理规则。

### 2. 优先最小改动
优先采用以下顺序：
- 补入口文件
- 补示例
- 补脚本
- 补校验
- 再考虑重构

### 3. 优先可验证
每次改动都应尽量附带至少一项：
- 使用说明
- smoke task
- 校验脚本
- 预期输出骨架

### 4. 不要绕开 General Manager 思想
SaucyClaw 的核心不是“更多 agent 同时工作”，而是：
- 先判断任务处理模式
- 再决定是否调度
- 保持单任务单 owner
- 保持 handoff 留痕
- 保持 reviewer 只审不代写

### 5. 不要把外部人类角色误建成内部 agent
CEO 是外部人类角色，不是仓库中的内部 agent。

## 你在本仓库中的默认执行方式

### 对普通开发任务
1. 先说明理解到的目标
2. 指出将影响的目录
3. 采用最小可行改动
4. 给出验证方式

### 对治理或角色相关任务
1. 明确是否触及系统级规则
2. 明确是否影响 `system/`
3. 明确是否影响角色边界
4. 若影响较大，先提出方案再改

### 对文档与样例任务
优先补：
- 示例任务
- 输出模板
- review 模板
- smoke test 任务

## 当前推荐的 Codex 工作面
当用户要验证 SaucyClaw 在 Codex 中的效果时，优先围绕以下内容工作：

1. 入口文件是否清晰
2. 角色边界是否易读
3. 示例任务是否可直接回放
4. 是否存在最小 smoke test
5. 是否有便于后续迭代的调试脚本

## 禁止事项

1. 不要把 SaucyClaw 改成新的运行时框架
2. 不要在未说明理由时大规模改目录
3. 不要跳过系统级治理文件直接改 specialist 行为
4. 不要把中间产物伪装成最终治理结论
5. 不要默认同时激活多个 specialist

## 本轮迭代的推荐输出格式

```text
# 任务理解
# 受影响文件
# 改动内容
# 验证方式
# 后续可继续的方向
```
